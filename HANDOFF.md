### ⚠️ Оговорка про чтение файлов (важно для следующей сессии)
Инструмент `web_fetch` при загрузке `application/game/gameStore.ts` (~17 КБ)
**регулярно «схлопывает» середину** — теряются 4 внутренних функции.
Пробовали 3 раза, всегда одно и то же. Поэтому любой рефакторинг этих функций
требует, чтобы пользователь **прислал их текст руками**. Не выдумывать их по
памяти — это заложит баги, которые «затвердеют» в тестах.

## 2. Куда идём и почему

### Цель
Превратить кодовую базу в честную **Clean Architecture** с **functional core** —
чтобы правила игры были чистыми, детерминированными и **тестируемыми без
браузера/DOM/рендера**.

### Слои (зависимости идут только внутрь)
```
presentation ──► application ──► domain ◄── (внешнего нет)
   (Pixi/React,      (сторы,            (чистые правила игры:
    DOM, ввод)        оркестрация)       бой, лут, прогрессия)
       │
       └──► infrastructure (RNG-seed, localStorage — технические детали)
```

**Правило зависимости:** `domain` ничего не знает о мире снаружи — ни о React,
ни о localStorage, ни о Pixi.

### Три практических выигрыша
1. **Правила игры — чистый код.** Бой/лут/leveling = чистые функции `(state, rng) → newState`.
2. **Любая деталь заменяема** без задевания игры (хранение, рендер, RNG).
3. **Тесты тривиальны и быстры** — функциональное ядро без побочных эффектов;
   сидируемый `RNG` даёт воспроизводимые забеги/тесты.

## 4. Что было сделано (детально)

Все правки доставлены **идемпотентными Node-скриптами** (а не патчами —
hand-written `.patch` оказался хрупким на подсчёте строк; скрипты через
`fs.writeFileSync`/`.replace()` надёжнее). Скрипты лежат в корне клона:

| Скрипт                  | Закрывает | Что делает |
|-------------------------|-----------|------------|
| `apply-refactor.mjs`    | #1, #2, #3a | `RNG` → `domain/services/rng.ts`; `createRandomSeed()` → `infrastructure`; правит 5 мест импорта `RNG`; `metaStore` → `createMetaStore(IMetaRepository)` + composition root `main.tsx`; числа боя/лута → `domain/config.ts`; `combatService` переписан |
| `apply-navigation.mjs`  | #4 | `presentation/navigation/navigationStore.ts` + `presentation/hooks/useCurrentScreen.ts`; `App.tsx` — экран производная от статуса (убран `useState`+`useEffect`-мост) |
| `apply-immer.mjs`       | #5 | `navigationStore`/`metaStore` → draft-мутации; `gameStore` обёрнут в `immer` (behaviour-preserving, zustand v5 shallow-merge по умолчанию) |
| `apply-cleanup.mjs`     | #6 | `STAT_BONUS_KEYS` + `addStatBonuses` через цикл; `characterFactory` переиспользует хелпер; `player` → `playerLayer` в `PixiGameCanvas`; `package.json` name |
| `apply-eslint-guard.mjs`| #7 | Поднимает ESLint с нуля (его не было!); flat-config; `no-restricted-imports` для 3 границ зависимостей + `lint`-скрипт; скрипт сам сканирует домен перед запуском |

### Как применять скрипты (строго в этом порядке, из корня клона)
```bash
git checkout -b refactor/clean-arch
node apply-refactor.mjs        # #1, #2, #3a
node apply-navigation.mjs      # #4
node apply-immer.mjs           # #5      (зависит от #2/#4!)
node apply-cleanup.mjs         # #6
node apply-eslint-guard.mjs    # #7
cd pixijs-roguelike-clean-architecture
npm install -D eslint @eslint/js typescript-eslint   # для #7
npm run build && npm run lint
```
Каждый скрипт в конце сам верифицирует результат и предупреждает, если
что-то не применилось.

## 5. Что предстоит сделать и почему

### 🔶 #3b — `turnService` (последний незакрытый пункт)
**Что:** вынести поток хода из «божественного» `gameStore` в чистую
тестируемую функцию `domain/services/turnService.ts` (`applyAction(input, action) → result`),
оставив стор тонкой обёрткой.

**Почему не сделано:** нужно перенести 4 функции из `gameStore.ts`, но
инструмент `web_fetch` **схлопнул их середину** (см. §1). Их текст нужно
попросить у пользователя.

**Нужны тексты ровно этих 4 функций:**
1. `handlePickup` — хвост ветки «предмет» (логика при полном инвентаре);
2. `runEnemyPhase` — **ИИ врагов** (агро, движение, атака игрока);
3. `finalizeTurn` — фиксация хода + **детекция гибели и начисление эссенции**;
4. `tickAbilityCooldown`.

**ВАЖНО — открытые вопросы из ревью (проверить в этих функциях):**
- `returnToHub()` сбрасывает стейт, **не** вызывая `applyRunResult`. Убедиться,
  что эссенция начисляется **ровно один раз** (по видимости — на экране
  `GameOverScreen`).
- Подтвердить, что `status: 'dead'` реально выставляется при гибели игрока
  (всё в навигации #4 держится на этом).
- Проверить корректность удаления убитого врага в `move` после удара.

**Почему это опционально, а не блокер:** 3b — это code smell «божественный стор»,
**НЕ нарушение чистой архитектуры**. Направление зависимостей и слои уже корректны.
3a (магические числа → config) сделана. Так что #3 можно считать «достаточно закрытым».

### 🧪 Тесты — следующий большой шаг (следующая сессия)
**Почему сейчас можно:** домен уже **по-настоящему чистый** (после #1–#7),
все правила игры — чистые функции. `turnService` для тестирования сервисов
**не нужен**. Тесты дадут ~80% ценности и создадут страховочную сеть на случае
будущего рефакторинга `gameStore`.

**Что ставить:** `vitest` (в `package.json` его нет — `npm i -D vitest`).

**Что покрывать (детерминированный сид `RNG`):**
- `combatService.resolveAttack` — крит, урон, дисперсия, `targetDied`;
- `statsService` — `computeEffectiveStats`/`applyDamage`/`applyHeal`/`spendMana` + кламп;
- `experienceService.applyXpGain` — мультилевел-ап, рост статов;
- `lootService.generateItem` — редкости по удаче, аффиксы, границы значений;
- `metaProgressionService` — расчёт эссенции и бонус-статов;
- `dungeonGenerator.generateFloor` — инварианты сетки, лестница, враги, боссы на каждом 5-м этаже.

Начать с **`combatService`** (самая ценная и изолированная).

## 6. Архитектурные правила, которые теперь закреплены

| Принцип | Где |
|---------|-----|
| Домен не импортирует внешние слои | #1 (`RNG` в домене) + #7 (ESLint guard физически не даст регрессить) |
| Dependency Inversion (порт + внедрение) | #2 (`IMetaRepository` → composition root `main.tsx`) |
| Functional core | чистые сервисы в `domain/services/*` |
| Single source of truth (навигация) | #4 (`useCurrentScreen` — производная статуса) |
| Иммутабельность через immer | #5 (3 стора) |
| DRY | #6 (`STAT_BONUS_KEYS`) |
